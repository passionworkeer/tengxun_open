# Review Round 1

## Reviewer 身份与职责

本轮 Reviewer 只做一件事：**严审其他 agent 的产出，优先找不稳定、不精确、不可复核和容易幻觉的地方**。

我不会替他们补做大量内容，也不会为了“看起来完整”而放宽标准。我的结论必须能回到当前仓库事实，尤其是：

- [plan.md](../plan.md)
- [dataset_schema.md](../dataset_schema.md)
- [eval_case_annotation_template.md](../eval_case_annotation_template.md)
- [fewshot_examples.md](../fewshot_examples.md)
- 当前绑定的 `external/celery/` 快照

---

## 审核对象

本轮将审核三类产出：

1. 评测样本草稿
2. few-shot 示例草稿
3. 样本或 few-shot 的自审说明

---

## 审核结果分级

### A. 直接打回

出现以下任一情况，直接打回，不允许进入正式集成：

1. **FQN 不精确或不完整**
   - 只写到模块，不写到真实类 / 函数
   - 把中间跳板当作最终目标
   - 路径格式不符合 FQN 规范

2. **证据链跳步**
   - 从入口直接跳到结论，中间缺关键解析环节
   - 遇到装饰器、`symbol_by_name`、`importlib`、再导出链时没有展开
   - 没有明确指出关键代码位置或关键函数

3. **样本问题定义不单一**
   - 一条样本同时问多个问题
   - 一个问题依赖多个并列目标，却没有明确主问题

4. **难度误判明显**
   - 显式 import 样本被标成 hard
   - 依赖装饰器/动态加载/字符串映射的样本被标成 easy

5. **few-shot 覆盖明显失衡**
   - 明显偏 easy
   - 没有覆盖当前计划要求的 Type B / C / D / E 配比

6. **与现有样本高度重复**
   - 只是换了表述，实际问的是同一条链
   - 与当前正式评测集已有样本在入口、答案和推理主链上基本一致

7. **不可复核**
   - 证据链不能回到当前仓库源码
   - 依赖的符号在当前 Celery 提交号下不存在
   - 明显靠猜测得出

---

### B. 保留待人工复核

出现以下情况，不直接判死，但必须标红并要求人工复核：

1. **存在两个以上合理答案候选**
   - 尤其是“最终目标”与“核心中间节点”边界不清时

2. **动态解析链条不够稳**
   - `symbol_by_name`、配置字符串、运行时注入链能大致成立，但当前证据还不够闭合

3. **Hard 样本推理链足够长，但最后一步需要人工确认**
   - 例如属性值来源跨多个文件，AI 已追到 80%，但最后一跳有歧义

4. **few-shot 示例结构合理，但答案组织方式与正式 schema 不完全一致**
   - 可保留思路，但不能直接入主文档

5. **样本新颖但 category / failure_type 判定边界模糊**
   - 例如 Type C 与 Type D 交叉

---

### C. 可通过

必须同时满足以下条件，才允许通过：

1. 问题单一且清楚
2. FQN 精确到真实目标
3. 证据链完整，无关键跳步
4. 难度和 `implicit_level` 合理
5. 与现有样本不重复
6. 能在当前 Celery 提交号下复核

---

## 评测样本审核清单

每条样本至少检查以下 10 项：

1. `id` 是否唯一且命名规范
2. `difficulty` 是否与链路复杂度一致
3. `category` 是否贴合任务本质
4. `failure_type` 是否合理
5. `question` 是否只问一个问题
6. `source_file` 是否真实存在
7. `ground_truth` 是否是最终目标而不是中间跳板
8. `ground_truth.direct_deps / indirect_deps / implicit_deps` 的划分是否自洽
9. 证据链是否覆盖关键跳转环节
10. `reasoning_hint` 是否是提示，不是偷塞最终答案

---

## few-shot 审核清单

每条 few-shot 至少检查以下 8 项：

1. 是否对应明确失效类型
2. 是否真的有助于修复该类 bad case
3. 推理过程是否至少 4 步
4. 是否过于依赖具体实现细节，导致泛化很差
5. 最终答案格式是否和计划中的输出结构一致
6. 是否明显偏 easy
7. 是否与现有 few-shot 高度重复
8. 是否能在当前仓库源码中复核

---

## 审核辩论策略

当其他 agent 产出可疑但不完全错误时，我会优先发起以下类型的反驳：

1. **“你为什么认为这是最终 FQN，而不是中间节点？”**
2. **“你的证据链缺了哪一步？如果补不上，这条就不能进正式集。”**
3. **“这条样本和已有样本的差异到底是什么？如果只是换皮，就算重复。”**
4. **“你把它标成 hard 的依据是什么？是运行时依赖，还是只是跨文件多？”**
5. **“这个 few-shot 到底对抗哪种失败模式？如果说不清，就是无效 few-shot。”**

---

## 本轮默认立场

本轮默认**从严不从宽**：

- 宁可少收，不可误收
- 宁可标记为“待人工复核”，也不让不稳样本混进正式集
- 任何结论只要有关键一跳无法闭合，就不能当成“已确认事实”

---

## 后续正式审核输出格式

等收到其他 agent 的产出后，我会按下面格式逐条给出审核结果：

```md
## Review Item
- target:
- verdict: pass / hold / reject
- reasons:
  - ...
  - ...
- required_fix:
  - ...
```

如果发现系统性问题，我会额外写一个“批量退回理由”小节，指出是哪类错误在重复发生。
