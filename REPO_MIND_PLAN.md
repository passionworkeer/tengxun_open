# RepoMind 优化项目计划

> **时间：** 2026.03 - 2026.05（12周）
> **目标：** 组内优秀 + 简历有东西写
> **核心原则：** 先分析再优化，每个决策有数据支撑

---

## 项目背景

**RepoMind** 是腾讯内部 MCP 工具，驱动代码变更分析（Changelog 生成 / Breaking Change 检测等）。

当前痛点：Changelog 生成 Pass Rate 仅 **~20%**，有大量优化空间。

---

## 两个核心目标

| 目标 | 衡量标准 |
|------|----------|
| **目标1：组里拿优秀** | 比同组竞争者的 Pass Rate 提升幅度更大，实验更严谨，报告更完整 |
| **目标2：简历有东西写** | 一句话能说清楚做了什么 + 数据提升了多少，面试官快速理解价值 |

**两个目标可同时满足，核心是产出有反直觉发现的结论。**

---

## 完整路线（12周）

### 第一阶段（第1-2周）：接入和诊断

**目标：搞清楚 RepoMind 在哪里失败**

```
具体工作：
1. 接入 RepoMind MCP，跑通一个完整调用
2. 选定分析方向（建议 Changelog 生成，基线最低）
3. 选定仓库（建议 Django，你最熟悉）
4. 批量跑 50 个失败 case
5. 人工分析失败原因，归纳成 5-8 个类别
```

**产出：失效模式分类报告**

```
失效模式分类报告
├── Type 1：工具未找到相关 commit（X%）
├── Type 2：找到 commit 但 Breaking Change 漏识别（X%）
├── Type 3：格式输出不规范（X%）
├── Type 4：多 commit 聚合逻辑错误（X%）
├── Type 5：跨文件变更关联断裂（X%）
├── Type 6：...（按实际发现补充）
└── Type 7：...

每类附 3 个具体 Bad Case
```

**明天开会要问的关键问题：**
> "我想从 Changelog 生成（当前 ~20%）入手做优化，能给我看一下这个任务上典型的失败 case 吗？我想先做失效模式分析，再设计优化方案。"

**先确认事项（影响工作深度）：**
- [ ] RepoMind 是内部 API 还是能看源码？ → 影响分析深度
- [ ] 当前 Changelog Pass Rate 数字确认
- [ ] 基线仓库选 Django 还是其他？

---

### 第二阶段（第3-5周）：PE 系统优化

**目标：逐层量化每个优化手段的贡献**

**单变量实验顺序：**
```
Baseline
→ + System Prompt 角色定义
→ + 输出格式约束
→ + CoT 推理引导
→ + Few-shot 示例库（≥20条）
→ + 后处理规则
每步单独记录 Pass Rate 变化
```

**CoT 模板示例（针对 Type 2 Breaking Change 漏检）：**
```
Step 1: 识别所有包含 API 变更的 commit
Step 2: 对每个变更检查：是否删除了公共接口？
Step 3: 是否修改了函数签名？
Step 4: 是否改变了返回值类型？
Step 5: 按 Breaking / Change / Fix 三类输出
```

**真正有技术含量的 PE 工作：**
- 不是换几句话
- 而是针对你发现的每类失效模式，设计专项 CoT 步骤

**产出：PE 方案文档**

```
PE方案文档
├── 四维度独立增益表（每步有数字）
├── 20条 Few-shot 示例库（按失效类型分布）
├── CoT 模板（针对不同任务类型）
└── 优化前后对比截图（Bad Case 变 Good Case）
```

---

### 第三阶段（第6-8周）：RAG 增强

**目标：解决 PE 无法解决的检索层问题**

**核心问题：LLM 不是因为不会推理才失败，而是根本没有看到正确的代码片段。**

**RAG 方案设计：**

```
Changelog 生成专用 RAG
├── 索引层：
│   ├── commit message 向量化
│   ├── diff 内容关键词索引
│   └── 文件变更图谱
├── 检索策略：
│   ├── 语义检索：找相似功能的变更
│   ├── 关键词检索：找精确的 API 名称
│   └── 图检索：找关联文件的变更
└── 上下文管理：
    ├── Token 预算控制
    └── Breaking Change 优先级排序
```

**产出：RAG 方案文档**

```
RAG方案文档
├── Pipeline 架构图
├── Recall@5 / MRR 检索指标
├── 端到端 Pass Rate 提升数据
└── RAG vs No-RAG 的 Hard case 对比
```

---

### 第四阶段（第9-10周）：跨语言泛化实验

**这是超越同组竞争者的关键。**

```
实验设计：
Django（Python）→ Spring Boot（Java）→ k8s（Go）

记录：
哪些优化策略跨语言有效？
哪些只在 Python 上有效？
为什么？
```

**可能发现的反直觉结论：**
```
"CoT 的 Step 2 在静态类型语言（Java/Go）上
  有效率比 Python 高 40%，
  因为静态类型让函数签名变化更可检测"
```

**产出：跨语言泛化报告**

```
跨语言泛化报告
├── 三语言 Pass Rate 对比表
├── 失效模式的语言差异分析
└── 普适性优化策略 vs 语言专项策略
```

## 赢过同组竞争者的关键结论形式

```
普通同学会做：
→ 调了 Prompt，Pass Rate 从 20% 到 40%
→ 报告写了优化前后对比

你要做的：
→ 先分析为什么失败（失效模式分类）
→ 针对每类失效设计专项方案
→ 每个方案有独立的增益数据
→ 在多个语言上验证泛化性
→ 找到一个反直觉的结论

最能赢的结论形式：
"大家以为问题是模型不够强，
 但我发现真正的瓶颈是工具调用顺序，
 换了调用顺序比换 GPT-5 效果提升更大"
```

## 风险与应对

| 风险 | 应对 |
|------|------|
| RepoMind 只能 API 调用，看不到源码 | 聚焦黑盒优化 + RAG 层介入 |
| 失败 case 归纳不出清晰的类别 | 先跑 20 个做预分类，再扩到 50 个验证 |
| 同组竞争者进度更快 | 聚焦"发现别人没发现的规律"，不拼速度 |
| 跨语言实验时间不够 | 前两个阶段压缩到第 8 周末完成，第 9-10 周专项跨语言 |
