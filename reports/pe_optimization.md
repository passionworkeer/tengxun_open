# Prompt Engineering 优化报告

> 模型：GPT-5.4 ｜ 评测集：50 条（Easy 15 / Medium 20 / Hard 15） ｜ 日期：2026-03-27

---

## 一、实验设计

严格单变量逐步叠加，不允许跳步：

```
Baseline → +System Prompt → +CoT → +Few-shot → +Post-processing
```

| 步骤 | 内容 | Token 增量 |
|:-----|:-----|:----------|
| Baseline | 零样本，仅 user 消息 + JSON 格式要求 | 基准 |
| +System Prompt | 角色定义 + 7 条格式约束（FQN 用 `.`、禁止 `:`） | +~5% |
| +CoT | 在 System Prompt 基础上加 5 步推理引导 | +~15% |
| +Few-shot | 在 CoT 基础上加 6 条动态选择的示例（从 20 条库中按相关性选） | +~40% |
| +Post-processing | 同 Few-shot 输入，输出经 FQN 校验 + 去重 + `:` → `.` 转换 | +~40% |

## 三、各步骤深度分析

### 3.1 System Prompt：单步增益最大（+0.1774）

**要点：**

- 对 Hard 提升最猛：0.1831 → 0.4210（**+129.8%**）
- 对 Medium：0.1785 → 0.3659（+105.0%）
- 角色定义 + 格式约束让模型输出更规范，减少了大量格式错误

**但有局限：**

- easy_012/easy_013 仍然 F1=0（模型坚持用 `:` 分隔符），仅靠 System Prompt 的文字约束不够
- 说明某些"格式习惯"需要更强的手段（few-shot / post-processing）才能纠正

### 3.2 CoT：反而小幅下降（-0.0130）

**这是本次实验最反直觉的发现。**

- Easy 从 0.5814 降到 0.5414（-0.0400）
- Hard 从 0.4210 降到 0.3832（-0.0378）
- 仅 Medium 微升 +0.0259

**CoT 导致回归的 15 个 case 中，典型案例如下：**

| Case ID | 难度 | 失效类型 | System Prompt F1 | CoT F1 | 回归幅度 |
|:--------|:----:|:--------:|:----------------:|:------:|:--------|
| easy_002 | easy | Type C | **1.0000** | 0.0000 | -1.0000 |
| celery_hard_121 | hard | Type A | 0.6250 | 0.1429 | -0.4821 |
| celery_hard_014 | hard | Type B | 0.4000 | 0.0000 | -0.4000 |
| medium_016 | medium | Type E | 0.4000 | 0.0000 | -0.4000 |
| easy_003 | easy | Type E | 0.7500 | 0.4000 | -0.3500 |
| easy_001 | easy | Type C | 0.6667 | 0.3333 | -0.3334 |
| easy_008 | easy | Type C | 1.0000 | 0.6667 | -0.3333 |

**归因分析：**

CoT 的推理引导让模型"想太多"。以 easy_002 为例，System Prompt 下模型直接给出正确答案，但 CoT 引导它"追踪再导出链"后反而绕了远路，输出了中间节点而非最终符号。

**结论：对 GPT-5.4 这种推理能力已经很强的模型，CoT 的固定推理模板是一种约束而非助力。** 如果要用 CoT，可能需要针对不同难度动态调整 CoT 的详细程度。

### 3.3 Few-shot：第二大增益（+0.1547）

**要点：**

- Medium 提升最显著：0.3918 → 0.6626（**+69.1%**）
- Hard：0.3832 → 0.5527（+44.2%）
- 20 条示例库按失效类型配比，模型能"照葫芦画瓢"

**但 Easy 反而微降 -0.0150**，因为 few-shot 示例多为中/高难度场景，对简单 case 引入了不必要的复杂性。

**典型提升案例：**

| Case ID | 难度 | 失效类型 | CoT F1 | Few-shot F1 | 提升 |
|:--------|:----:|:--------:|:------:|:-----------:|:----|
| medium_011 | medium | Type C | 0.0000 | **1.0000** | +1.0000 |
| medium_002 | medium | Type E | 0.5714 | **1.0000** | +0.4286 |
| medium_004 | medium | Type C | 0.6667 | **1.0000** | +0.3333 |
| medium_015 | medium | Type D | 0.5000 | **1.0000** | +0.5000 |

### 3.4 Post-processing：稳健增益（+0.0343）

**要点：**

- 全面提升，无任何回落
- 格式净化（FQN 校验 + 去重 + `:` → `.` 自动转换）持续稳定贡献
- easy_012/easy_013 在此阶段从 0.0 → 0.4000，后处理确实修了格式问题

## 四、按失效类型的 PE 效果

| 失效类型 | Cases | Baseline | SP | CoT | Few-shot | PostProc | PE 最大增益 |
|:---------|:-----:|:--------:|:--:|:---:|:--------:|:--------:|:----------|
| **Type A** (长上下文截断) | 4 | 0.1950 | 0.4529 | 0.3488 | 0.5094 | **0.6173** | +0.4223 |
| **Type B** (隐式依赖幻觉) | 12 | 0.2091 | 0.3601 | 0.3440 | **0.5254** | 0.5140 | +0.3163 |
| **Type C** (再导出链断裂) | 12 | **0.5972** | 0.6389 | 0.6556 | 0.5972 | 0.6445 | +0.0584 |
| **Type D** (命名空间混淆) | 5 | 0.0000 | 0.4505 | 0.4905 | **0.5886** | 0.5619 | +0.5886 |
| **Type E** (动态加载失配) | 17 | 0.1781 | 0.3708 | 0.3449 | 0.6462 | **0.7041** | +0.5260 |

**关键发现：**

1. **Type E 和 Type D 是 PE 最能解决的**（+0.53 和 +0.59），因为它们有大量格式/模式问题可以通过示例和约束修复
2. **Type C 本身基线就高（0.60）**，PE 增益有限，因为再导出链是 GPT-5.4 已经擅长的
3. **Type B 有提升但天花板明显**（0.21 → 0.53），幻觉问题 PE 无法根治，需要 FT

## 六、典型 Bad Case 对比

### Case 1：easy_012（Type E — 格式问题）

| 阶段 | F1 | 模型预测 |
|:-----|:--:|:--------|
| Gold | — | `celery.concurrency.eventlet.TaskPool` |
| Baseline | 0.0000 | `celery.concurrency.eventlet:TaskPool`（`:` 错误） |
| System Prompt | 0.0000 | `celery.concurrency.eventlet:TaskPool`（仍用 `:`） |
| CoT | 0.0000 | 同上，还多输出了中间路径 |
| **Few-shot** | **0.4000** | `celery.concurrency.eventlet.TaskPool` ✓ + 多余依赖 |
| **Post-processing** | **0.4000** | 同上，后处理自动去重 |

**结论**：System Prompt 文字约束不够，few-shot 示例示范了正确格式才修复。

### Case 2：easy_002（Type C — CoT 导致回归）

| 阶段 | F1 | 模型预测 |
|:-----|:--:|:--------|
| Gold | — | `celery.app.shared_task` |
| System Prompt | **1.0000** | `celery.app.shared_task` ✓ |
| CoT | 0.0000 | 模型开始追踪中间再导出节点，输出了错误路径 |

**结论**：本来 System Prompt 直接答对了，CoT 的"分步推理"反而引入了干扰。

### Case 3：medium_002（Type E — Few-shot 从 0 到满分）

| 阶段 | F1 | 模型预测 |
|:-----|:--:|:--------|
| Baseline | 0.0000 | 模型输出文件路径而非 FQN |
| System Prompt | 0.0000 | 格式对了但解析终点错误 |
| CoT | 0.5714 | 推理有所改善但不完整 |
| **Few-shot** | **1.0000** | 完全正确 — 示例教会了模型什么是正确的解析终点 |

**结论**：Few-shot 对 Type E 动态加载场景效果最好，因为模型需要看到"正确答案长什么样"。

## 七、核心结论

### 7.1 五条关键发现

1. **System Prompt 是 ROI 最高的单步**：仅加角色定义和格式约束，Avg F1 从 0.27 跳到 0.45（+65.8%），对 Hard 场景提升近 130%
2. **Few-shot 是绝对量最大的增益来源**：在 SP 基础上再 +0.15，Medium 翻倍、Type E 从 0.37 到 0.65
3. **CoT 对 GPT-5.4 效果为负**：15 个 case 回归，主要因为 GPT-5.4 推理能力强，固定推理模板反而约束了它
4. **Post-processing 是稳定的兜底**：不做推理改变，只做格式净化，每步都有正增益
5. **全量 PE 从 0.27 → 0.62（+131%）**：效果显著但仍有上限，Type B 幻觉问题需要 FT 解决

### 7.2 工程落地建议

- **默认推荐**：System Prompt + Few-shot + Post-processing（跳过 CoT），Avg F1 约 0.62
- **如果 Token 预算紧张**：仅 System Prompt，以 +5% Token 换 +65% F1
- **CoT 的使用场景**：可能对推理能力较弱的模型（如 Qwen3.5-9B）有效，待验证

### 7.3 下一步

1. 用 GLM-5 和 Qwen3.5-9B 重跑 PE 实验，验证 CoT 的负效果是否只在 GPT-5.4 上出现
2. PE + RAG 联合实验
3. 针对 Type B（幻觉）启动 FT 训练
