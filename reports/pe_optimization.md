# Prompt Engineering 优化报告

模型：`GPT-5.4`  
评测集：`data/eval_cases.json`（正式 `54-case`）  
实验路径：`baseline -> system_prompt -> cot -> fewshot -> postprocess`

## 1. 实验目标

本实验不引入 RAG，也不改模型权重，只测试 Prompt Engineering 对代码依赖分析任务的独立贡献：

- `System Prompt`
- `CoT`
- `Few-shot`
- `Post-processing`

目标不是追求单一最终分数，而是量化每一步到底帮了什么、帮了多少。

## 3. 核心结论

### 3.1 最终收益

- `baseline 0.2745 -> postprocess 0.6062`
- 绝对提升：`+0.3317`
- 相对提升：`+120.8%`

### 3.2 各阶段角色

1. `System Prompt` 负责把输出从“会说一些对的东西”收敛到“按 JSON/FQN 口径回答”。
2. `CoT` 在正式 54-case 结果上是正收益，说明对这批问题来说，显式推理步骤能帮助模型分层 direct / indirect / implicit。
3. `Few-shot` 是最大的单步增益来源，说明模型不是不会做，而是不稳定；示例能显著减少解题口径漂移。
4. `Post-process` 是低成本兜底，主要修格式、重复项和边界脏输出。

## 4. 分难度增益

### 4.1 从 baseline 到最终版

| Difficulty | Baseline | Final | Delta |
|------|------:|------:|------:|
| easy | 0.3907 | 0.6651 | +0.2744 |
| medium | 0.2602 | 0.6165 | +0.3563 |
| hard | 0.2010 | 0.5522 | +0.3512 |

### 4.2 解读

- `medium` 和 `hard` 的绝对收益最高，说明 PE 不只是修格式，更是在帮助模型稳定完成链路归约。
- `easy` 也提升明显，但这不是最主要价值所在。

## 5. 按失效类型看 PE 到底修了什么

| Failure Type | Baseline | System Prompt | CoT | Few-shot | Post-process |
|------|------:|------:|------:|------:|------:|
| Type A | 0.1875 | 0.1732 | 0.4607 | 0.5256 | 0.5133 |
| Type B | 0.2539 | 0.2515 | 0.3501 | 0.5354 | 0.5502 |
| Type C | 0.3939 | 0.5320 | 0.5364 | 0.6333 | 0.7394 |
| Type D | 0.2121 | 0.1990 | 0.4032 | 0.5636 | 0.6159 |
| Type E | 0.2851 | 0.3394 | 0.3790 | 0.5807 | 0.5801 |

### 5.1 结论

1. **Type D / Type E 是 PE 最能修的两类问题**  
   它们都依赖“输出口径 + 答题模式”的稳定性。Few-shot 和后处理能显著降低模型把相关文件、字符串别名和最终 FQN 混在一起的概率。

2. **Type C 在正式结果里也显著受益**  
   这和早期 50-case 草稿里的印象不同。说明在更完整的 54-case 口径下，PE 对 re-export / proxy 链也有稳定帮助。

3. **Type A 的主要收益来自 CoT 之后**  
   这类问题要求模型显式区分“谁负责注册 / 谁负责执行 / 哪一步真正 materialize”，CoT 明显有帮助。

## 6. 四个阶段分别贡献了什么

### 6.1 System Prompt

`0.2745 -> 0.3138`

作用：

- 统一输出格式
- 限制解释性废话
- 强化 direct / indirect / implicit 的层级区分

局限：

- 只能解决“回答方式不稳定”的问题
- 对动态 alias / finalize 链这种需要真正推理的 case 帮助有限

### 6.2 CoT

`0.3138 -> 0.4218`

这是本轮正式结果里最值得注意的变化，因为它和早期 50-case 草稿结论不同。  
在当前正式数据上，CoT 对 `Type A / Type D / Type B` 都带来可观增益，说明“显式推理模板”在正式集上是有帮助的。

但它不是纯正收益：

- `13` 个 case 相比 `system_prompt` 发生回退
- 典型回退集中在一些本来就比较短、模型本可直接答对的问题上

所以 CoT 的正确定位不是“永远开”，而是：

- 对中高复杂度问题有帮助
- 对非常简单的问题可能引入额外干扰

### 6.3 Few-shot

`0.4218 -> 0.5733`

这是最大的单步增益：

- 对 `Type E`：`0.3790 -> 0.5807`
- 对 `Type D`：`0.4032 -> 0.5636`
- 对 `Type B`：`0.3501 -> 0.5354`

说明 Few-shot 的价值不是“告诉模型新知识”，而是：

- 告诉模型最终答案长什么样
- 告诉模型何时该停在最终符号，而不是继续往上追
- 告诉模型如何区分 direct / indirect / implicit

### 6.4 Post-process

`0.5733 -> 0.6062`

这是一个典型的“低成本高确定性”模块：

- 增益不如 Few-shot 大
- 但几乎不依赖模型能力
- 对真实部署非常重要，因为能稳定去掉一些边界脏输出

## 7. 代表性解释

### 7.1 为什么 Few-shot 这么关键

这个任务不是普通问答，而是**结构化依赖归约**。模型往往“知道一些相关东西”，但不知道：

- 哪个是最终目标
- 哪些应该放在 `direct`
- 哪些只能算 `implicit`

Few-shot 的作用就是提供“评分口径模板”。

### 7.2 为什么 Post-process 仍然必要

即使模型已经理解题目，仍可能出现：

- 重复输出
- FQN 与文件路径混用
- direct / indirect 放错层

这些问题不应该再交给模型二次思考，而应该交给程序化后处理。

## 8. 最终建议

### 8.1 当前最优 PE 组合

在正式 `54-case` 上，建议默认使用：

- `System Prompt`
- `CoT`
- `Few-shot`
- `Post-process`

即当前 `postprocess` 版本。

### 8.2 工程建议

1. 如果只追求最高准确率，不要去掉 CoT。
2. 如果后续要给 Qwen 做同类 PE 实验，优先复用这套四阶段结构。
3. 后处理必须保留，因为这是最便宜、最稳定的线上增益来源。

## 9. 结论

这轮正式结果说明：

- PE 不是“锦上添花”，而是当前项目中最强的独立优化手段。
- 对代码分析类任务，**Few-shot + 明确层级输出 + 程序化后处理** 是最核心的三件事。
- 后续无论 Qwen 做不做 RAG、做不做 FT，PE 都应该作为默认基础层存在。
