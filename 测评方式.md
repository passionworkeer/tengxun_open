## 1. 端到端功能验证（Fail-to-Pass）

| 论文             | 发表                       | 核心贡献                                                     | 链接                                                         |
| ---------------- | -------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **SWE-bench**    | **ICLR 2024**（Princeton） | 首创 Fail-to-Pass 评测范式，2,294 个真实 GitHub Issue，用 PR 合入前后测试状态变化作为 ground truth | [arxiv:2310.06770](https://arxiv.org/abs/2310.06770)         |
| **EvoCodeBench** | 北大 2024                  | 函数体替换为 pass → LLM 生成 → 跑仓库自带测试，275 个样本持续演进 | [arxiv:2404.00599](https://arxiv.org/abs/2404.00599)         |
| **BeyondSWE**    | arXiv 2026.03              | 扩展到跨仓库/依赖迁移/全仓库生成，500 个实例 4 种设置，端到端验证 | [arxiv:2603.03194](https://arxiv.org/abs/2603.03194)         |
| **RealBench**    | **FSE 2026**（武大+北大）  | UML + 自然语言需求驱动的仓库级代码生成端到端评测             | [FSE 2026](https://conf.researchr.org/details/fse-2026/fse-2026-research-papers/187/) |
| **SWE-Factory**  | **FSE 2026**               | 全自动 Issue 解决数据构建流水线，扩展 Fail-to-Pass 到多语言  | [arxiv:2506.10954](https://arxiv.org/abs/2506.10954)         |

**权威性**：SWE-bench 是 ICLR 2024 论文，目前是业界评测 AI Coder 的**事实标准**，所有主流 AI Coder（Copilot/Cursor/Claude Code/OpenHands）都在该榜单上竞争。

## *2. 过程质量评测（Context Retrieval / Process-Oriented）*

| 论文                 | 发表                             | 核心贡献                                                     | 链接                                                 |
| -------------------- | -------------------------------- | ------------------------------------------------------------ | ---------------------------------------------------- |
| **ContextBench**     | arXiv 2026.02                    | 首个面向过程的上下文检索评测，1,136 个任务 + 人工标注 gold context，提出上下文召回率/精确率指标 | [arxiv:2602.05892](https://arxiv.org/abs/2602.05892) |
| **SWE-ContextBench** | arXiv 2026.02                    | 评测 Agent 跨任务的上下文重用能力，1,100 基础任务 + 376 关联任务 | [arxiv:2602.08316](https://arxiv.org/abs/2602.08316) |
| **CodeCompass**      | arXiv 2026.02                    | 提出"导航悖论"，258 次试验对比图导航 vs 文本检索的过程差异   | [arxiv:2602.20048](https://arxiv.org/abs/2602.20048) |
| **CrossCodeEval**    | **NeurIPS 2023**（Columbia+AWS） | 用静态分析自动筛选必须跨文件上下文才能补全的样本，评估检索质量 | [arxiv:2310.11248](https://arxiv.org/abs/2310.11248) |

**权威性**：CrossCodeEval 是 NeurIPS 2023 论文。ContextBench 虽然目前是 arXiv 预印本，但它提出的"Agent 苦涩教训"（复杂脚手架对检索质量提升微小）已被广泛引用，是 2026 年上下文评测的标杆。

| 论文/工具                  | 发表                                 | 核心贡献                                                     | 链接                                                         |
| -------------------------- | ------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Asleep at the Keyboard** | **IEEE S&P 2023** (CCS系列)          | 首个系统研究 AI 生成代码安全性，发现 12.1% 含漏洞，涵盖 77 种 CWE | [arxiv:2311.04356](https://arxiv.org/abs/2311.04356)         |

| **Beyond Human-Readable**  | arXiv 2026.04                        | 受控实验对比 4 种格式的 Token 经济，发现压缩悖论             | [arxiv:2604.07502](https://arxiv.org/abs/2604.07502)         |

## 4. 人工评审 / LLM-as-Judge

| 论文                 | 发表                                  | 核心贡献                                                     | 链接                                                        |
| -------------------- | ------------------------------------- | ------------------------------------------------------------ | ----------------------------------------------------------- |
| **CodeAgentBench**   | **ACL 2024**（北大）                  | 101 个样本，tree-sitter 静态分析 + 两人标注（600 人时）+ 手动补充测试 | [ACL 2024](https://aclanthology.org/2024.acl-long.737/)     |
| **DevEval**          | **ACL 2024 Findings**                 | 手工标注的代码生成 benchmark，对齐真实仓库，人工评估代码质量 | [ACL 2024](https://aclanthology.org/2024.findings-acl.214/) |
| **Codified Context** | arXiv 2026.02                         | 283 个开发会话的实证观察 + 4 个案例研究，定性评估上下文基础设施效果 | [arxiv:2602.20478](https://arxiv.org/abs/2602.20478)        |
| **DDI 调试衰减模型** | **Nature Computational Science 2025** | 提出调试效率指数衰减模型 E(t)=E0*e^(-λt)，基于实验数据拟合   | Nature CompSci 2025                                         |

**权威性**：CodeAgentBench 是 ACL 2024 主会论文。DDI 模型发表在 Nature Computational Science，是调试衰减领域的权威依据。

## 总结：每种方式的论文支撑力度

| 评测方式       | 最高规格论文                                                 | 权威性评价                                                |
| -------------- | ------------------------------------------------------------ | --------------------------------------------------------- |
| 端到端功能验证 | **ICLR 2024** (SWE-bench), **FSE 2026** (RealBench)          | **最强**，SWE-bench 是业界事实标准                        |
| 过程质量评测   | **NeurIPS 2023** (CrossCodeEval), arXiv 2026 (ContextBench)  | **较强**，CrossCodeEval 顶会，ContextBench 是 2026 新范式 |
| 静态分析/Lint  | **IEEE S&P 2023** (Asleep at the Keyboard), **ICLR 2026 Workshop** | **强**，安全顶会背书                                      |
| 人工/LLM 评审  | **ACL 2024** (CodeAgentBench), **Nature CompSci 2025** (DDI) | **强**，但可复现性最差                                    |

四种方式都有顶会/顶刊论文支撑，不存在哪种"没有依据"的情况。之前建议的**混合方式**本质上就是论文们自己在做的事——SWE-bench 用端到端，ContextBench 补过程评测，Asleep at the Keyboard 用静态扫描，CodeAgentBench 用人工评审——只是各自侧重不同维度。
