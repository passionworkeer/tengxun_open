●项目 ：基于微调/PE/RAG 的代码分析领域效果优化
考核题目【LLM 代码分析瓶颈诊断】选定一个代码分析方向（调用链追踪、历史变更理解、跨文件依赖分析、语义检索等），在真实开源项目上构建 ≥50 条评测用例，评估当前主流 LLM 的基线表现，诊断低分场景的共性瓶颈（如长上下文丢失、隐式依赖忽略、跨文件关联断裂等）
●【Prompt Engineering 系统优化】系统性地优化提示策略——System Prompt 角色定义与输出格式约束、Few-shot 示例库构建（≥20 条）、Chain-of-Thought 推理引导设计、输出后处理规则（过滤/排序/去重），量化每项优化的效果增益
●【RAG 增强管线构建】构建面向代码分析的 RAG Pipeline——代码片段向量化索引（Embedding 选型调优）、语义+关键词混合检索策略、上下文窗口管理、检索结果与生成模型的融合策略，评估 RAG 增强对分析效果的提升
●【模型微调实验】构建代码分析领域的微调数据集（≥500 条），采用 LoRA/QLoRA 等方法对开源 LLM 进行领域适配训练，监控过拟合，评估微调效果
●【消融实验与策略选择】设计完整的消融实验矩阵（PE only / RAG only / Fine-tune only / PE+RAG / PE+Fine-tune / All），识别效果最优的策略组合及其适用边界
●低难度：选择一个代码分析方向，构建 ≥50 条评测用例，通过 Prompt Engineering（System Prompt + Few-shot + CoT + 后处理）进行优化，量化优化前后的效果差异，输出 PE 方案文档和对比报告
●中难度：在 PE 基础上，构建 RAG Pipeline + 微调数据集（≥500 条）并完成 LoRA 微调，进行完整消融实验，输出各策略独立及组合效果的量化对比报告
●验收要求及评价指标瓶颈诊断质量：评测用例质量：用例是否基于真实项目、有明确的正确答案、覆盖不同难度
●瓶颈识别准确性：是否精准定位到低分场景的共性特征（而非泛泛描述）
●数据支撑：分析结论是否有充分的评测数据和错误样本佐证
●PE 方案质量：优化系统性：是否覆盖 System Prompt / Few-shot / CoT / 后处理四个维度
●效果量化：每项优化是否有独立的效果数据，而非仅给出最终总分
●方案可复现性：Prompt 方案文档是否详尽到他人可直接复用
●RAG 方案质量：Pipeline 完整度：向量索引 → 检索 → 上下文管理 → 融合生成，各环节是否完整
●检索精度：检索结果的相关性评分（Recall@K、MRR 等）
●端到端效果：RAG 增强后的整体分数提升幅度
●消融实验与整体评价：实验设计严谨性：是否包含单一手段和组合策略的完整消融矩阵
●最优策略识别：是否明确给出效果最优的策略组合及其适用条件
●可复现性：是否提供完整实验代码、配置、脚本及 README
